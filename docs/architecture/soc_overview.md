# SoC Overview

The system is based on a MicroBlaze MCS soft-core processor subsystem (comprising a MicroBlaze CPU and an I/O module) integrated into a custom FPGA-based SoC architecture.

The processor communicates with internal hardware modules through a vendor proprietary IP bus (MCS I/O Bus) .

A bridge module converts the processor-side bus transactions into a simplified internal MMIO-oriented bus used by the peripheral subsystem.

The SoC is divided into two main subsystems:

- MMIO subsystem
- VideoCore subsystem

## MMIO Subsystem

The MMIO subsystem manages memory-mapped peripherals such as:

- UART
- System timer
- GPIO output
- GPIO input

An MMIO controller decodes addresses and routes read/write transactions to the selected peripheral slot.

The architecture supports up to 64 peripheral slots, each exposing up to 32 memory-mapped registers of 32 bits.

## VideoCore Subsystem

The VideoCore subsystem will be dedicated to video generation and display-related processing modules.

## MCS I/O Bus Signals

The MicroBlaze MCS communicates with the SoC peripherals through the MCS I/O Bus. The MCS-to-MMIO bridge converts these signals into the internal MMIO bus used by the peripheral subsystem.

### Address Signals

| Signal             | Direction      | Description                                                      |
| ------------------ | -------------- | ---------------------------------------------------------------- |
| `io_address[31:0]` | Master → Slave | 32-bit byte-addressable address generated by the MicroBlaze MCS. |

The bridge converts the 32-bit byte address into the internal MMIO word address by discarding the two least significant bits (`address[1:0]`), since all MMIO accesses are aligned 32-bit transactions.

---

### Data Signals

| Signal                | Direction      | Description                               |
| --------------------- | -------------- | ----------------------------------------- |
| `io_write_data[31:0]` | Master → Slave | Data written by the processor.            |
| `io_read_data[31:0]`  | Slave → Master | Data returned by the selected peripheral. |

The bridge forwards write data directly to the MMIO subsystem and routes peripheral read data back to the processor.

---

### Control Signals

| Signal                | Direction      | Description                                                            |
| --------------------- | -------------- | ---------------------------------------------------------------------- |
| `io_write_strobe`     | Master → Slave | Indicates a write transaction.                                         |
| `io_read_strobe`      | Master → Slave | Indicates a read transaction.                                          |
| `io_addr_strobe`      | Master → Slave | Indicates that the address on `io_address` is valid.                   |
| `io_byte_enable[3:0]` | Master → Slave | Selects which bytes of the 32-bit word participate in the transaction. |
| `io_ready`            | Slave → Master | Handshake signal indicating transaction completion.                    |

---

## Signals Used in This Design

The bridge actively uses:

* `io_address`
* `io_write_data`
* `io_read_data`
* `io_write_strobe`
* `io_read_strobe`

These signals are sufficient to implement memory-mapped register accesses to all MMIO peripherals.

---

## Signals Not Used

### io_addr_strobe

The `io_addr_strobe` signal is intended to indicate that the address presented by the processor is valid.

In this design it is not used because the MicroBlaze MCS always asserts valid addresses whenever a read or write transaction is issued. Therefore the bridge assumes that any transaction accompanied by `io_read_strobe` or `io_write_strobe` already contains a valid address.

For this reason no additional address validation logic is required.

---

### io_byte_enable

The `io_byte_enable[3:0]` signal allows byte-level accesses inside a 32-bit word.

Examples:

| io_byte_enable | Meaning                |
| -------------- | ---------------------- |
| 1111           | write/read all 4 bytes |
| 0001           | access only byte 0     |
| 0011           | access bytes 0 and 1   |

In this project all peripherals expose 32-bit memory-mapped registers and software always performs aligned word accesses through MMIO helper functions.

As a result:

```c
io_write(base, offset, value);
io_read(base, offset);
```

always generate full 32-bit transactions.

Therefore `io_byte_enable` is ignored by the bridge.

---

### io_ready

The MCS I/O bus supports a handshake mechanism through `io_ready`.

A peripheral may hold `io_ready = 0` while it is processing a transaction and assert `io_ready = 1` when the operation is complete.

In this design all MMIO peripherals complete read and write operations within a single clock cycle.

Therefore the bridge permanently asserts:

```vhdl
io_ready <= '1';
```

and no wait-state or handshake logic is required.

This simplifies both the bridge implementation and the peripheral interface.
